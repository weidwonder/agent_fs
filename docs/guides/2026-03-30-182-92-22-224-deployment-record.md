## 6. 风险与后续建议

- 当前实际对外端口已切换为 `1202`。
- 最终验证结果：
  - `curl http://127.0.0.1:1202/health` 返回 `{"status":"ok"}`
  - `curl http://182.92.22.224:1202/health` 返回 `{"status":"ok"}`
- 根因定位：
  - `mihomo` 透明代理接管了主机的策略路由与透明转发，导致公网入站连接虽然能完成 TCP 建连，但 HTTP 请求无法正常到达应用层。
  - 停止 `mihomo` 后，公网访问立即恢复正常，因此该问题不是项目代码、容器端口映射或阿里云安全组本身导致。
- 当前处理：
  - 已停止 `mihomo.service`
  - 已执行 `systemctl disable mihomo`，当前状态为 `disabled + inactive`，服务器重启后不会自动恢复
- 当前服务器无法稳定直连公网仓库，不适合依赖在线拉镜像与在线下载模型
- 若后续继续使用该大陆服务器，建议至少补齐以下一项：
  - 调整或移除 `mihomo` 的透明代理策略，避免影响 Docker 与公网入站流量
  - 建立公司内网镜像仓库
  - 将业务镜像构建与分发固定到 CI/CD
- 若后续要切到域名访问：
  - 先完成 ICP 备案
  - 再接入 Nginx + HTTPS 证书

## 7. 2026-03-31 二次排障记录

### 7.1 文件一直 `pending`

现象：

- 用户注册、创建项目并上传 `.docx` 后，文件状态长期停留在 `pending`

根因：

- 服务端调用 `boss.send('index-file', ...)` 前没有显式创建 `index-file` 队列
- `pg-boss` 未自动创建该业务队列，导致上传后任务静默未入队

处理：

- 在 `server` 与 `worker` 启动时显式执行 `boss.createQueue('index-file')`
- 在入队逻辑中校验 `boss.send()` 返回值，避免再次静默失败

### 7.2 `.docx` 转换仍失败

第一次修复后，文件状态开始流转，但两个 `.docx` 均转为 `failed`，错误为 `DocxConverter 请求超时`。

已定位出的两层根因：

1. 转换器启动目标解析错误
   - Linux 容器内同时存在 `DocxConverter.dll` 与同名可执行文件
   - 旧逻辑会先把 `.dll` 路径解析成同名可执行文件，最终错误执行原生二进制
   - 已修复为：
     - 非 Windows 平台优先保留 `.dll`
     - 仅当 `.dll` 不存在时才回退到可执行文件

2. 运行时缺少 .NET 依赖
   - 实际转换时容器缺少 `libicu`
   - 已在生产镜像中补充 `libicu72`

### 7.3 镜像构建链路补强

另一个隐藏问题是：

- Docker 构建阶段之前没有重新编译 `plugin-docx` / `plugin-excel` 的 TypeScript
- 如果本地 `dist` 过旧，镜像会把旧逻辑直接打进去

已处理：

- 在 `docker/Dockerfile` 中增加：
  - `pnpm --filter @agent-fs/plugin-docx exec tsc`
  - `pnpm --filter @agent-fs/plugin-excel exec tsc`

这样镜像内会重新生成这两个插件的 `dist`，不再依赖宿主机陈旧产物。

### 7.4 当前最后状态

- 最新镜像已重新推送到阿里云容器镜像仓库
- 最新镜像已在远端重启
- 远端容器内已验证：
  - `resolveConverterPath()` 返回 `.dll`
  - `resolveConverterLaunchTarget()` 返回 `dotnet + DocxConverter.dll`
- 重新入队两个 `.docx` 后，远端主机随后出现异常：
  - 公网 `http://182.92.22.224:1202/health` 返回 `Empty reply from server`
  - `22` 端口可连通，但 `sshd` 在密钥交换前即关闭连接

结论：

- “任务一直 `pending`” 的根因已经修复
